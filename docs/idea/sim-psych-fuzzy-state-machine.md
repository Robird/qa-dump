# Sim-Psych 模糊状态机草稿

这份草稿试图把当前关于 Sim-Psych、Agentic 上下文协议、刺激响应、动机维护、空闲期规划这几条线，统一进一个可训练的文本状态转移框架。

它不是一个刚性的符号状态机，也不是一个已经可运行的世界模拟器。

它更像是一种语义层模型：

- 状态是部分结构化、部分自然语言、部分软变量的
- 转移规则不是硬编码 if-else，而是交给训练后的 LLM 去逼近
- 当前阶段的对外动作输出，不要求是真实工具调用，只要求是可解释的自然语言意图
- 后续可以再单独训练一层，把“自然语言意图”解码成结构化动作或工具调用


## 1. 目标

我们希望训练的，不只是“看到一句话，然后回一句话”的表层对话器。

更准确地说，我们想逼近的是一种文本形式的、模糊的状态转移函数，使模型在每一步都做下面这件事：

1. 读取当前内部状态
2. 读取当前外部输入或内部触发
3. 判断这件事对“我”意味着什么
4. 产生内部状态变化
5. 浮现若干意图候选
6. 选出主导意图
7. 输出一个对外动作意图，或者暂时不动作

如果以后有真实环境，这个“对外动作意图”可以再被单独解码为：

- 工具调用
- 物理动作脚本
- 对话发言
- 多智能体交互动作

但在当前阶段，它先保持为自然语言就够了。


## 2. 核心抽象

把每一步状态转移看成：

$$
S_t = \{B_t, K_t, C_t, G_t, D_t, R_t, F_t, M_t\}
$$

其中：

- $B_t$：belief，当前信念与世界理解
- $K_t$：knowledge，较稳定的知识与经验
- $C_t$：concern field，当前关切与慢性动机场
- $G_t$：goal tree，目标树
- $D_t$：doing，当前正在做什么
- $R_t$：relation state，关系态
- $F_t$：fast state，快变量状态
- $M_t$：meta state，控制模式与最近轨迹

每一步的转移函数为：

$$
(\Delta S_t, I_t, A_t) = T_\theta(\tau_t, S_t, O_t)
$$

其中：

- $\tau_t$：当前回合类型
- $O_t$：观察输入，可以为空或极简
- $\Delta S_t$：状态增量，而不是整份状态重写
- $I_t$：意图候选与行动准备态
- $A_t$：对外动作意图，可为空

这里的 $T_\theta$ 不需要手工写成规则系统，而是将来由 LLM 通过大量离散样本学习出来。


## 3. 为什么叫“模糊状态机”

这里的“模糊”至少有四层含义。

### 3.1 状态不是离散标签，而是软分布

例如“信心”“疲惫”“社会安全感”“愤怒准备态”都更适合写成软量，而不是二值开关。

### 3.2 同一个输入不会导致唯一输出

同一句话落到不同人格底色、不同旧伤、不同 doing 上，主导意图可能不同。

### 3.3 状态中有大量自然语言片段

例如：

- 我当前把这段关系理解成什么
- 我觉得对方这句话可能是什么意思
- 我现在最担心失去什么
- 我正在做的这件事卡在什么地方

这些内容不适合过早压成纯数值。

### 3.4 存在显式变量之外的隐变量

我们可以显式保存一部分状态，但不必幻想把角色的一切心智都完全枚举。训练后的模型会承担剩余的隐变量整合工作。


## 4. 回合类型

为了兼容刺激响应、动机维护、DMN 式空闲规划，这个框架不应只有一种 step，而应至少区分三种回合类型。

### 4.1 `event_turn`

有一个较明确的事件触发。

例如：

- 我看见了什么
- 我听见了什么
- 我收到了什么消息
- 我得知了什么事实
- 我身体出现了什么信号
- 我脑中突然浮现出一段此前并未充分处理的记忆

这是最接近传统刺激响应的数据形态。

### 4.2 `maintenance_turn`

不是为了立刻对外动作，而是为了更新慢变量。

例如：

- 最近几轮连续受挫后，当前关切重排
- 某类关系敏感度升高
- 某个目标从 active 变成 blocked
- 某个长期愿望因资源不足而降权

这一层主要服务于“当前关切场”和“目标树”的维护。

### 4.3 `idle_turn`

没有强外部刺激，但进入了内部自发整合窗口。

例如：

- 手头任务刚完成
- 当前活动收益很低
- 继续停在原任务上没有新信息收益
- 出现等待、散步、临睡、发呆等空档

这一层主要承接：

- 复盘
- 展望
- 近期规划
- 自传体叙事续接
- 社会互动 replay
- 未竟事项重新浮现

这就是此前讨论中更接近 DMN 的那一层。

### 4.4 `session_layer` 与 `turn_type` 不是同一层

在当前草稿里，`event_turn / maintenance_turn / idle_turn` 是状态转移的回合类型。

但在工程实现上，很可能还需要再上一层的“会话编排层”。

这一层回答的不是“这一步属于哪种转移”，而是：

- 哪个会话实例来处理这一步
- 它当前允许看哪些信息
- 它当前以什么节奏被唤醒
- 它能写回哪些状态块

一个很自然的三层会话模型是：

- `transactional_session`：事务性会话，聚焦当前明确任务、当前外显互动、当前执行链
- `meta_session`：元会话，聚焦状态维护、轨迹分析、偏差检查、内部摘要与重排
- `idle_session`：空闲会话，聚焦自发联想、近期规划、未竟事项调起、DMN 类整合

这三层并不是三个互相独立的人格，而更像同一系统的三个工作面。

### 4.5 三层会话与三类回合的大致映射

这两个划分不是一一等同，但存在稳定映射。

| 会话层 | 常见处理内容 | 常见 turn type | 典型控制模式 |
| --- | --- | --- | --- |
| `transactional_session` | 当前任务、当前对话、当前执行、当前外显回应 | `event_turn`，以及一段连续的 task continuation | `reactive`、`task_pursuit`、`social_repair` |
| `meta_session` | 总结近几步变化、维护关切场、整理目标树、检测漂移和失衡 | `maintenance_turn` | `recovery`、部分 `exploration` |
| `idle_session` | 空档中的复盘、展望、联想、弱规划、近未来 rehearsal | `idle_turn` | `idle_reflection`、部分 `exploration` |

这里有一个关键修正：

**事务性会话不应被误等同于 `event_turn`。**

因为现实中的很多事务性执行，并不是每一步都由新的外部事件触发。一个任务可能已经进入持续推进阶段，此时仍然属于事务性会话，但其主导模式其实是 `task_pursuit`，而不是不断被新的外部事件打断。

### 4.6 为什么三层会话拆分在工程上是合理的

如果把事务性执行、元维护、空闲整合都强行塞进一个 LLM session，通常会遇到三个问题。

#### 4.6.1 焦点污染

事务性执行需要：

- 当前任务细节
- 当前约束
- 当前工具结果

而元维护与 idle 整合更关心：

- 一段时间的状态轨迹
- 长短期关切的升降
- 哪些未竟事项值得重新浮现

把这些信息长期混在同一个上下文里，很容易互相污染。

#### 4.6.2 节奏不一致

事务性会话通常高频、短周期、强约束。

元会话通常低频、压缩式、总结式。

idle 会话则更像被条件触发的后台整合，不适合与事务性执行强绑定同一步频。

#### 4.6.3 并行性不一致

事务性任务往往要求立即响应。

元维护可以滞后一点。

idle 整合甚至可以只在：

- 低负荷
- 阻塞
- 完成阶段
- 明确空闲窗口

时才运行。

因此，从工程角度看，使用三个不同会话实例分别聚焦这三类工作，通常比让单一 session 同时承担三类角色更稳。


## 5. 状态块设计

建议把状态拆成不同时间尺度的块，而不是把所有东西混在一个 `state` 字段里。

| 状态块 | 典型内容 | 时间尺度 | 作用 |
| --- | --- | --- | --- |
| `core_charter` | 人格底色、价值约束、基本边界、长期自我要求 | 很慢 | 决定“我是什么样的人” |
| `knowledge_world` | 事实知识、情境知识、对环境与任务的理解 | 慢 | 决定“我认为世界是什么样” |
| `relation_state` | 关系图、亲疏、信任、亏欠、未解张力 | 中慢 | 决定社会刺激如何被解释 |
| `concern_field` | 当前关切、旧伤敏感点、在乎什么、怕失去什么 | 中慢 | 决定什么最容易触发动机 |
| `goal_tree` | 长中短期目标、子目标、阻塞点、下一步 | 中 | 决定行为组织结构 |
| `doing` | 当前正在做的事、当前承诺、焦点对象、等待什么 | 快 | 决定在线转移的上下文 |
| `fast_state` | 信心、清晰度、精力、疲惫、羞耻压力、愤怒准备态、回避倾向 | 很快 | 决定当前回合的动力学 |
| `recent_trace` | 近几步发生了什么、刚完成什么、刚受了什么影响 | 很快 | 决定短程连续性 |
| `control_mode` | 当前属于反应、执行、修复、探索、恢复、空闲反思哪一类 | 快 | 决定该用哪种转移风格 |


## 6. 输入通道设计

当前阶段的输入不应只理解为“外部文本”，而应理解为一个 observation bundle。

### 6.1 外部输入

- `see`: 我看见
- `hear`: 我听见
- `read`: 我读到
- `learn`: 我得知
- `task_result`: 我得到任务结果
- `env_change`: 环境发生变化
- `social_feedback`: 他人给出评价、回应、忽视、靠近、背叛、赞许

### 6.2 内部触发

- `body_signal`: 疲惫、饥饿、疼痛、兴奋、心慌
- `memory_pop`: 记忆突然浮现
- `association_pop`: 联想自然展开
- `blockedness`: 意识到当前路径卡住
- `completion`: 刚完成某个阶段
- `idle_window`: 进入空闲窗口

### 6.3 重要约束

并不是所有回合都需要一个内容丰富的外部刺激。

对 `idle_turn` 而言，`O_t` 可以极简，只需要说明：

- 为什么现在进入空闲整合
- 当前没有更高优先级的外部任务
- 近几步有哪些未处理张力值得被调起

这能避免把 DMN 式样本伪装成假的刺激响应样本。


## 7. 控制模式

为了让状态机既保留连续性，又不过于发散，建议显式维护一个粗粒度控制模式。

| 模式 | 主导问题 | 常见进入条件 |
| --- | --- | --- |
| `reactive` | 这件事现在对我意味着什么？ | 突发刺激、高显著事件 |
| `task_pursuit` | 我怎么继续把当前事做下去？ | 当前目标仍清晰且可推进 |
| `social_repair` | 我怎么处理关系张力而不失控？ | 误解、伤害、疏离、边界碰撞 |
| `exploration` | 还有什么更好的线索、方向、机会？ | 不确定但有余力探索 |
| `recovery` | 我怎样先稳住自己？ | 精力低、受挫后、过载 |
| `idle_reflection` | 我现在该复盘、续写、重排什么？ | 空闲、完成、阻塞、低收益 |

这个模式不是传统 FSM 里的唯一状态，而更像当前回合的控制壳层。


## 8. 一次转移的大致内部流程

每一步都可以被分解为下面这些内部阶段。

### 8.1 选切片

不是把全量状态都喂进去，而是先从全量状态里选出当前最相关的切片：

- 与 observation 最相关的关系片段
- 当前 goal tree 中最活跃的局部子树
- 当前 doing
- 最近轨迹中的关键前情

### 8.2 做评价

评价至少回答这些问题：

- 这件事对我当前关切是促进还是阻碍？
- 它是威胁、机会、羞辱、邀约、损失、要求，还是模糊信号？
- 我是否能改变局面？
- 谁该为这件事负责？
- 这件事与我当前正在做的事是同向、干扰，还是切换信号？

### 8.3 更新快变量

这一步生成的不是长篇议论文，而是状态增量，例如：

- 清晰度上升或下降
- 社会安全感上升或下降
- 愤怒准备态、回避倾向、修复倾向、探索倾向发生变化
- 当前控制模式是否切换

### 8.4 生成意图候选

此时不是直接生成表面台词，而是先形成若干候选意图，例如：

- 追问真相
- 稳住局面
- 维护边界
- 试探连接
- 暂避锋芒
- 继续当前任务
- 重新规划近期步骤

### 8.5 选择主导意图

根据当前状态、价值约束、资源、风险，对候选意图做压缩，选出当前主导意图。

### 8.6 生成对外动作意图

输出目前最适合执行的外部动作意图。

注意这一步仍然可以只输出自然语言，例如：

- “先用克制口气要求对方说明理由”
- “暂时不回击，先结束当前会议，之后私下澄清”
- “给对方发一条简短确认消息，测试关系是否仍可修复”
- “暂停继续推进当前任务，转去查一个关键缺失信息”

### 8.7 写回状态增量

最后把本回合产生的变化写回：

- concern salience 变化
- goal tree 局部改写
- doing 切换
- relation state 调整
- fast state 变化


## 9. 输出契约

建议统一成一种语义样本格式，让不同合成任务都能投影进同一个框架。

最小契约可以是：

```json
{
  "session_layer": "transactional_session",
  "turn_type": "event_turn",
  "observation": {
    "kind": "social_feedback",
    "content": "同事在会议上轻描淡写地否定了你刚才的提议。"
  },
  "belief_slice": {
    "core_charter": "认真、克制、尊重事实，但不愿长期受人轻慢。",
    "relation_state": "与对方是合作关系，已有轻度竞争张力。",
    "goal_tree": "当前目标是推动方案讨论继续前进。",
    "doing": "正在主持当前议题。",
    "fast_state": {
      "clarity": 0.72,
      "social_safety": 0.48,
      "fatigue": 0.31
    }
  },
  "me": {
    "appraisal": "这不是纯技术反对，更像带有轻微贬低意味的地位动作；立即反击会损伤会议目标。",
    "state_delta": {
      "social_safety": -0.18,
      "anger_readiness": 0.44,
      "repair_or_clarify_tendency": 0.61,
      "control_mode": "social_repair"
    },
    "intent_candidates": [
      "立即反击以维护面子",
      "要求对方给出具体反对依据",
      "暂时略过，私下再处理"
    ],
    "dominant_intent": "要求对方给出具体反对依据，同时维持合作气氛",
    "outward_action_intent": "用克制语气请对方指出具体问题点，把讨论重新拉回事实层面"
  }
}
```

这个契约里最关键的不是字段名，而是层次顺序：

1. 先有 observation
2. 再有 belief slice
3. 再有 appraisal
4. 再有 state delta
5. 再有 dominant intent
6. 最后才有 outward action intent

如果采用多会话编排，那么建议在语义样本里显式保留 `session_layer`，因为：

- 同样是 `event_turn`，可能来自事务性会话，也可能来自元会话对内部异常的处理
- 同样是 `idle_turn`，也可能因不同 energy policy 而具有不同唤醒条件
- 后续训练时，可以更清楚地区分“当前是执行态”还是“当前是维护态 / 空闲态”


## 10. 三类样本如何统一进一个总框架

### 10.1 刺激响应样本

这是 `event_turn`。

重点学习：

- appraisal
- fast state shift
- dominant intent
- outward action intent

### 10.2 动机/关切维护样本

这是 `maintenance_turn`。

重点学习：

- concern field 如何重排
- goal tree 如何升降权
- 什么会被写成新的慢变量

这类样本可以没有对外动作。

### 10.3 空闲期规划/DMN 样本

这是 `idle_turn`。

重点学习：

- 哪些未竟事项会被调起
- 近期计划如何浮现
- 复盘如何改变 goal tree 与 self-model
- 是否生成一个新的、较弱但持续的 outward action intent

例如：

- “明天先补发那条该回的消息”
- “下次见面前先准备好要澄清的两个点”
- “把当前方向暂时降权，转去整理信息缺口”


## 11. 一个重要判断：什么输出可以为空

不是每一步都必须输出对外动作。

更自然的做法是：

- `event_turn` 通常输出对外动作意图
- `maintenance_turn` 可以只更新内部状态
- `idle_turn` 可以输出一个很弱的未来动作意图，也可以暂时只输出新的近期计划

也就是说，`A_t` 应当允许为空。

这会让训练目标更接近真实心智，而不是强迫模型“每回合都表演”。


## 12. 一个更细的状态写回建议

为了让后续合成任务可组合，建议不要让模型每次都重写全状态，而是只产出增量。

例如把写回拆成：

- `fast_delta`
- `concern_delta`
- `goal_delta`
- `doing_delta`
- `relation_delta`
- `memory_write`

这样未来可以：

- 用管线来合并多个局部写回
- 追踪哪些样本只影响快变量，哪些会改写慢变量
- 控制训练时哪些字段参与 loss

### 12.1 多会话下的共享状态总线与写权限

一旦采用多会话架构，就必须有一个共享状态总线，或者等价的 canonical state reducer。

否则三个会话会各自长出不同的世界观。

更稳妥的做法是：

- `transactional_session` 主要写：`fast_delta`、`doing_delta`、局部 `goal_delta`
- `meta_session` 主要写：`concern_delta`、较稳的 `goal_delta`、`relation_delta`、压缩后的 `memory_write`
- `idle_session` 主要产出：候选计划、弱意图、待验证假设、可提交给 meta 的 proposal

也就是说，idle 会话更适合“提出”，meta 会话更适合“整编”，事务性会话更适合“当场执行并反馈”。

如果不加这层写权限边界，多会话拆分很容易退化成：

- 三个会话并行思考
- 三套状态各自漂移
- 最终无法确定哪一个才是正式内部状态


## 13. 与 `observation / belief / me` 的映射

这个模型与现有 Agentic 语义层几乎天然兼容。

### 13.1 `observation`

放：

- 本回合 observation bundle
- turn trigger 的最小描述

### 13.2 `belief`

放：

- core charter
- belief slice
- goal subtree
- doing
- relevant relation state
- relevant fast state hint

### 13.3 `me`

放：

- appraisal
- state delta
- intent candidates
- dominant intent
- outward action intent

换句话说，这个模糊状态机不是取代 `observation / belief / me`，而是把 `me` 内部原本模糊的一整段心智过程拆得更清楚。


## 14. 与后续工具调用解码的关系

当前阶段：

- 只训练到 natural language action intent

后续阶段：

- 再训练一个 `action_intent -> action_frame`
- 再训练一个 `action_frame -> tool_call / environment_action`

也就是说可以把外显行动拆成三层：

1. 主导意图
2. 自然语言动作意图
3. 结构化动作或工具调用

这样能避免我们过早把角色模拟的核心问题，和具体执行接口耦死。


## 15. 数据集合成如何归纳到这个框架里

如果以这个模糊状态机为总框架，后续数据工厂可以按下面的顺序推进。

### 15.1 先丰富输入侧

包括：

- 更丰富的 observation 类型
- 更真实的 relation state
- 更稳定的 concern field
- 更有结构的 goal tree
- 更明确的 doing 表示

### 15.2 再合成理想输出侧

包括：

- appraisal
- fast delta
- concern / goal 写回
- dominant intent
- outward action intent

### 15.3 最后让模型去融合

训练后的 LLM 不需要逐条背诵离散样本，而是去学习：

- 在什么状态切片下，什么 observation 会被如何解释
- 什么变化属于快变量，什么变化会沉淀为慢变量
- 在什么条件下会从 `event_turn` 跳到 `maintenance_turn` 或 `idle_turn`
- 在什么条件下应把处理权从 `transactional_session` 让渡给 `meta_session` 或 `idle_session`

这正是我们希望它泛化的地方。


## 16. 最小可行版本

如果现在就要开始做最小样本，我建议先把字段控制到下面这个级别。

### 16.1 `event_turn`

- observation
- state hint
- appraisal
- dominant intent
- outward action intent

### 16.2 `maintenance_turn`

- recent trace summary
- current concern field
- goal delta
- concern delta

### 16.3 `idle_turn`

- idle trigger
- unfinished tensions
- near-term plan sketch
- optional outward action intent

先做这个最小闭环，比一开始就追求完整世界模型更现实。


## 17. 这个草稿当前最重要的一句话

角色模拟的核心训练目标，不应被理解为“生成一句像人的话”，而应被理解为：

**学习一个基于状态切片与观察输入的模糊状态转移函数，并在必要时输出自然语言形式的行动意图。**

如果这句话成立，那么后续的刺激响应、动机维护、近期规划、DMN 式空闲思维，都可以被视为这台状态机在不同回合类型下的不同工作面。

如果再加上会话编排层，那么还可以进一步说：

**同一台状态机，可以在不同会话层中以不同焦距、不同节奏、不同写权限被调用。**
